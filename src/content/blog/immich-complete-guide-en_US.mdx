## Why Self-Host Your Photos?

Three reasons, in order of importance:

1. **Privacy**. Your photos are scanned by cloud AI — for ads, for indexing, for purposes you didn't consent to. Self-hosted means the only entity reading your photos is you.

2. **Cost**. Google One 2TB = $100/year. iCloud+ 2TB = $100/year. That's $800 over the life of a phone. A one-time $200 hard drive runs forever.

3. **No vendor lock-in**. Google killed Picasa. Google killed Google Photos unlimited storage. When they kill the next thing, your data goes with it.

## Installation

### Step 1: Create the project directory

```bash
mkdir -p ~/immich && cd ~/immich
```

### Step 2: Download the environment file

```bash
curl -o .env https://raw.githubusercontent.com/immich-app/immich/main/docker/example.env
```

**Critical edit:** Change at least these values in `.env`:

```bash
# ponytail: change these or regret it later
DB_PASSWORD=your-strong-password-here
UPLOAD_LOCATION=/path/to/your/photos  # where your media lives
```

### Step 3: Download Docker Compose

```bash
curl -o docker-compose.yml https://raw.githubusercontent.com/immich-app/immich/main/docker/docker-compose.yml
```

### Step 4: Start it

```bash
docker compose up -d
```

First boot takes 2-3 minutes (pulling images, initializing the database). Watch with:

```bash
docker compose logs -f
```

Visit `http://your-server:2283` — that's it, you're running.

### Full docker-compose.yml (Annotated)

Here's the compose file with every variable explained:

```yaml
# No version key needed — Docker Compose v2+ ignores it
name: immich

services:
  # Main app server: web UI, API, photo/video serving
  immich-server:
    container_name: immich_server
    image: ghcr.io/immich-app/immich-server:release
    volumes:
      - ${UPLOAD_LOCATION}:/usr/src/app/upload  # your photo/video storage
      - /etc/localtime:/etc/localtime:ro          # sync container clock to host
    env_file:
      - .env
    ports:
      - 2283:2283                                 # web UI port
    depends_on:
      - database
      - redis
    restart: always

  # ML models: face detection, CLIP smart search, duplicate detection
  immich-machine-learning:
    container_name: immich_machine_learning
    image: ghcr.io/immich-app/immich-machine-learning:release
    volumes:
      - model-cache:/cache                        # persisted model weights
    env_file:
      - .env
    restart: always

  # Postgres with pgvector for ML embeddings and metadata
  database:
    container_name: immich_postgres
    image: tensorchord/pgvecto-rs:pg14-v0.2.1
    environment:
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: immich
      POSTGRES_USER: ${DB_USERNAME}
      POSTGRES_INITDB_ARGS: '--data-checksums'
    volumes:
      - pgdata:/var/lib/postgresql/data           # database files
    env_file:
      - .env
    restart: always

  # Redis: caching, job queues, session management
  redis:
    container_name: immich_redis
    image: redis:7-alpine
    healthcheck:
      test: redis-cli ping || exit 1
    restart: always

volumes:
  pgdata:        # postgres data
  model-cache:   # ml model weights
```

> **Pitfall #1:** The `pgvecto-rs` image is ARM-incompatible on some setups. If you're on Raspberry Pi or ARM64, use `tensorchord/pgvecto-rs:pg14-v0.2.1` — check the Immich docs for the latest ARM-compatible image.

## Configuration Deep Dive

### Environment Variables That Actually Matter

| Variable | Default | What It Does | My Recommendation |
|----------|---------|--------------|-------------------|
| `DB_PASSWORD` | (required) | Postgres password | Use a strong random string |
| `UPLOAD_LOCATION` | (required) | Where photos are stored | Dedicated drive or mount point |
| `JWT_SECRET` | (auto-generated) | Auth token signing | Leave as-is unless you know what you're doing |
| `IMMICH_MACHINE_LEARNING_URL` | `http://immich-machine-learning:3003` | ML endpoint | Don't change unless custom ML setup |
| `UPLOAD_LIMIT` | `300` | Max upload size (MB) | Increase for 4K+ video: `10000` |

### Storage Layout

After a few weeks, your upload directory looks like this:

```
/upload/
├── library/          # Original files, organized by user
├── thumbnail/        # Generated thumbnails (various sizes)
├── encoded-video/    # Transcoded video files
├── upload/           # Upload staging area
└── sidecar/          # EXIF metadata files
```

**Rule of thumb:** Budget 1.5x your raw media size for total storage (thumbnails + ML data + originals).

## First-Time Setup

1. Open `http://your-server:2283`
2. Create admin account
3. **Enable ML features** (Settings → Machine Learning):
   - Face Detection: Enabled
   - CLIP Embedding: Enabled for smart search
   - Duplicate Detection: Enabled
4. Set up storage template (Settings → Storage Template):

```
{{y}}/{{MM}}/{{filename}}
```

This organizes uploads into `2026/06/photo.jpg` — clean and predictable.

## Mobile App Setup

### Install

- **iOS:** [App Store](https://apps.apple.com/app/immich/id1613945686)
- **Android:** [Google Play](https://play.google.com/store/apps/details?id=app.alextran.immich) or [F-Droid](https://f-droid.org/packages/app.alextran.immich/)

### Connect

1. Open app → Settings → Server
2. Enter `http://your-server:2283` (or `https://` if behind reverse proxy)
3. Log in with your admin account
4. Enable **Background Backup** (Settings → Backup):
   - Select albums to backup
   - Set upload size limit
   - Enable **WiFi-only** upload to save mobile data

### First Sync Expectations

- First upload of 10,000 photos takes 2-8 hours depending on network
- App stays in foreground during initial sync (iOS limitation)
- After initial sync, new photos appear within seconds on WiFi

> **Pitfall #2:** iOS background upload is unreliable. Apple throttles background app refresh. Best practice: open the app once a day, let it catch up.

## Machine Learning

Immich uses several ML models ([official docs](https://immich.app/docs/features/ml)). Here's what each does:

### Face Detection & Recognition

- **Model:** InsightFace
- **What it does:** Finds faces, groups them by person, lets you name them
- **RAM:** ~2GB during processing
- **Accuracy:** ~90% for clear faces, lower for groups/side profiles

### CLIP Embedding

- **Model:** OpenCLIP
- **What it does:** Enables "smart search" — find photos by description, e.g., "sunset at the beach with a dog"
- **RAM:** ~4GB during processing
- **Speed:** Processes ~100 images/minute on modern hardware

### Duplicate Detection

- **What it does:** Finds visually similar photos, lets you clean up
- **Best used:** After importing from multiple sources

### Disable ML (If Low on Resources)

```yaml
# In docker-compose.yml, remove or disable:
# immich-machine-learning service
```

Or set `IMMICH_MACHINE_LEARNING_ENABLED=false` in `.env`. You lose face recognition and smart search, but everything else works.

## Performance Tuning

### For Large Libraries (50K+ photos)

```bash
# Increase Postgres shared buffers
# Add to database command in docker-compose.yml:
command: postgres -c shared_buffers=512MB -c effective_cache_size=2GB
```

### For Slow Uploads

```bash
# Increase upload limit (in .env)
UPLOAD_LIMIT=10000  # 10GB for 4K video
```

### For Low-RAM Systems (4GB)

- Disable machine learning (`IMMICH_MACHINE_LEARNING_ENABLED=false`)
- Use thumbnail generation only (disable CLIP)
- Set Postgres to 256MB shared_buffers

### Monitoring

```bash
# Check resource usage
docker stats

# Check logs
docker compose logs -f immich_server
docker compose logs -f immich_machine_learning
```

## Common Pitfalls & Fixes

### 1. "Upload Failed" on Mobile

**Cause:** Default upload limit is 300MB. 4K videos exceed this.

**Fix:** Set `UPLOAD_LIMIT=10000` in `.env`, restart with `docker compose restart`.

### 2. Faces Not Detected

**Cause:** ML container not running or out of RAM.

**Fix:**
```bash
docker compose ps  # Check ML container status
docker compose logs immich_machine_learning  # Check for OOM errors
```

If OOM: allocate more RAM or disable ML temporarily.

### 3. External Library Not Showing

**Cause:** Volume mount is wrong.

**Fix:** Ensure the path in `docker-compose.yml` matches your actual storage:
```yaml
volumes:
  - /mnt/photos:/usr/src/app/upload
```

And the path must be accessible by the container user.

### 4. PostgreSQL Connection Refused

**Cause:** Database container failed to start (usually password mismatch).

**Fix:** Check `.env` matches what Postgres expects. If corrupted:
```bash
docker compose down -v  # WARNING: deletes database
docker compose up -d    # Fresh start
```

### 5. Thumbnails Look Blurry

**Cause:** Low-quality thumbnail generation.

**Fix:** Not a bug — Immich generates multiple sizes. The "blurry" preview loads first, full quality loads after. Check Settings → Thumbnail Configuration.

## Backup Strategy

**3-2-1 Rule:** 3 copies, 2 different media, 1 offsite.

```bash
# Simple rsync backup to external drive
rsync -av --progress /path/to/immich/upload/ /mnt/backup/immich/

# Automated with cron (daily at 3am)
0 3 * * * rsync -av /path/to/immich/upload/ /mnt/backup/immich/
```

For database backup:
```bash
# $DB_USERNAME comes from your .env (default: postgres)
docker exec immich_postgres pg_dump -U "$DB_USERNAME" immich > immich_db_backup.sql
```

> **Pitfall #3:** Don't just copy the database — copy the `/upload` directory too. The database is metadata; your actual photos are in `/upload`.

## FAQ

### Can I migrate from Google Photos to Immich?

Yes. Use [Immich CLI](https://github.com/immich-app/immich-cli) to upload via API, or use the mobile app to re-upload from your phone (which already has the photos). There's no automatic Google Photos import — you'll need to export from Google Takeout first.

### Does Immich support RAW photos?

Yes. It processes RAW files and generates previews. Lightroom DNG, Canon CR2/CR3, Sony ARW, Nikon NEF — all supported.

### Can I share albums with non-Immich users?

Yes. Public shared links with optional password protection. No account required for viewing.

### What's the difference between Immich and PhotoPrism?

| | Immich | PhotoPrism |
|--|--------|------------|
| **Mobile App** | Excellent (native) | Good (PWA) |
| **Auto-Backup** | Yes (background) | No (manual) |
| **ML Models** | More (faces, CLIP, duplicates) | Fewer (faces only) |
| **UI** | Google Photos-like | More desktop-oriented |
| **Community** | 60K+ stars | 35K+ stars |

**My take:** Immich if you want a Google Photos replacement. PhotoPrism if you want a photo library manager.

### How much RAM do I really need?

- **4GB:** Disable ML, still works great for storage and viewing
- **8GB:** Full ML features, smooth for 50K+ photos
- **16GB+:** Overkill unless you're running 100K+ photos with all ML enabled

### Can I use Immich behind a reverse proxy?

Yes. See the [official reverse proxy docs](https://immich.app/docs/install/reverse-proxy). Works with Nginx, Caddy, Traefik, and Coolify.

### Which should I choose: Immich vs PhotoPrism vs Lychee?

| Use Case | Recommendation |
|----------|---------------|
| **Google Photos replacement** | **Immich** — native mobile backup, smart search, face recognition |
| **Photo library on low-resource hardware** | **Lychee** — runs on 512MB RAM, dead simple |
| **Photo library manager (desktop-focused)** | **PhotoPrism** — clean UI, good metadata tools |
| **Already running Nextcloud** | **Nextcloud Photos** — no extra stack needed |
| **Privacy-first, minimal features** | **Lychee** — smallest attack surface |

**TL;DR:** Immich is the most feature-complete. PhotoPrism is the most balanced. Lychee is the lightest. If you already have Nextcloud, use its photos app before adding another service.
