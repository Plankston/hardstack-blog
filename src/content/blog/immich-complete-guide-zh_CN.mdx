## 为什么要自托管照片？

三个原因，按重要性排序：

1. **隐私**。你的照片被云端 AI 扫描——用于广告、用于索引、用于你根本没同意的用途。自托管意味着唯一读取你照片的实体就是你自己。

2. **成本**。Google One 2TB = $100/年。iCloud+ 2TB = $100/年。手机生命周期内就是 $800。一块 $200 的硬盘能跑一辈子。

3. **没有厂商锁定**。Google 砍了 Picasa。Google 砍了 Google Photos 无限存储。等他们砍下一个的时候，你的数据一起没了。

## 安装部署

### 第一步：创建项目目录

```bash
mkdir -p ~/immich && cd ~/immich
```

### 第二步：下载环境变量文件

```bash
curl -o .env https://raw.githubusercontent.com/immich-app/immich/main/docker/example.env
```

**必须改的配置：** 至少把 `.env` 里这些值改了：

```bash
# ponytail: 不改的话后面会后悔
DB_PASSWORD=your-strong-password-here
UPLOAD_LOCATION=/path/to/your/photos  # 你的照片存储路径
```

### 第三步：下载 Docker Compose 文件

```bash
curl -o docker-compose.yml https://raw.githubusercontent.com/immich-app/immich/main/docker/docker-compose.yml
```

### 第四步：启动

```bash
docker compose up -d
```

首次启动需要 2-3 分钟（拉取镜像、初始化数据库）。用这个命令看日志：

```bash
docker compose logs -f
```

访问 `http://你的服务器IP:2283`——搞定，你已经跑起来了。

### 完整 docker-compose.yml（逐行注释）

下面是每个变量都解释清楚的 compose 文件：

```yaml
# 不需要 version 字段——Docker Compose v2+ 已经忽略它了
name: immich

services:
  # 主服务：Web UI、API、照片/视频服务
  immich-server:
    container_name: immich_server
    image: ghcr.io/immich-app/immich-server:release
    volumes:
      - ${UPLOAD_LOCATION}:/usr/src/app/upload  # 你的照片/视频存储
      - /etc/localtime:/etc/localtime:ro          # 容器时钟同步到宿主机
    env_file:
      - .env
    ports:
      - 2283:2283                                 # Web UI 端口
    depends_on:
      - database
      - redis
    restart: always

  # ML 模型：人脸检测、CLIP 智能搜索、重复检测
  immich-machine-learning:
    container_name: immich_machine_learning
    image: ghcr.io/immich-app/immich-machine-learning:release
    volumes:
      - model-cache:/cache                        # 持久化模型权重
    env_file:
      - .env
    restart: always

  # Postgres + pgvector，用于 ML 嵌入向量和元数据
  database:
    container_name: immich_postgres
    image: tensorchord/pgvecto-rs:pg14-v0.2.1
    environment:
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: immich
      POSTGRES_USER: ${DB_USERNAME}
      POSTGRES_INITDB_ARGS: '--data-checksums'
    volumes:
      - pgdata:/var/lib/postgresql/data           # 数据库文件
    env_file:
      - .env
    restart: always

  # Redis：缓存、任务队列、会话管理
  redis:
    container_name: immich_redis
    image: redis:7-alpine
    healthcheck:
      test: redis-cli ping || exit 1
    restart: always

volumes:
  pgdata:        # postgres 数据
  model-cache:   # ml 模型权重
```

> **坑 #1：** `pgvecto-rs` 镜像在某些 ARM 环境下不兼容。如果你用树莓派或 ARM64，用 `tensorchord/pgvecto-rs:pg14-v0.2.1`——去 Immich 官方文档查最新的 ARM 兼容镜像。

## 配置详解

### 真正重要的环境变量

| 变量 | 默认值 | 作用 | 我的建议 |
|----------|---------|--------------|-------------------|
| `DB_PASSWORD` | （必填） | Postgres 密码 | 用一个强随机字符串 |
| `UPLOAD_LOCATION` | （必填） | 照片存储位置 | 专用磁盘或挂载点 |
| `JWT_SECRET` | （自动生成） | 认证令牌签名 | 不知道在干嘛就别动 |
| `IMMICH_MACHINE_LEARNING_URL` | `http://immich-machine-learning:3003` | ML 服务地址 | 不自定义 ML 就别改 |
| `UPLOAD_LIMIT` | `300` | 最大上传大小（MB） | 4K 视频用户改成 `10000` |

### 存储目录结构

用几周之后，你的上传目录长这样：

```
/upload/
├── library/          # 原始文件，按用户组织
├── thumbnail/        # 生成的缩略图（多种尺寸）
├── encoded-video/    # 转码后的视频文件
├── upload/           # 上传暂存区
└── sidecar/          # EXIF 元数据文件
```

**经验法则：** 总存储预算 = 原始媒体大小 × 1.5（缩略图 + ML 数据 + 原始文件）。

## 首次设置

1. 打开 `http://你的服务器IP:2283`
2. 创建管理员账号
3. **开启 ML 功能**（设置 → 机器学习）：
   - 人脸检测：开启
   - CLIP 嵌入：开启（智能搜索用）
   - 重复检测：开启
4. 设置存储模板（设置 → 存储模板）：

```
{{y}}/{{MM}}/{{filename}}
```

这样上传的文件会自动组织成 `2026/06/photo.jpg`——干净、可预测。

## 机器学习功能

Immich 使用了多个 ML 模型（[官方文档](https://immich.app/docs/features/ml)）。下面逐个说：

### 人脸检测与识别

- **模型：** InsightFace
- **作用：** 找到人脸，按人物分组，支持命名
- **内存：** 处理时约 2GB
- **准确率：** 清晰人脸约 90%，合照/侧脸会低一些

### CLIP 嵌入

- **模型：** OpenCLIP
- **作用：** 启用"智能搜索"——用描述找照片，比如"海滩上的日落和一只狗"
- **内存：** 处理时约 4GB
- **速度：** 现代硬件上约 100 张/分钟

### 重复检测

- **作用：** 找到视觉相似的照片，方便清理
- **最佳使用场景：** 从多个来源导入后

### 关闭 ML（内存不够时）

```yaml
# 在 docker-compose.yml 里，移除或禁用：
# immich-machine-learning 服务
```

或者在 `.env` 里设置 `IMMICH_MACHINE_LEARNING_ENABLED=false`。你会失去人脸识别和智能搜索，但其他功能都正常。

## 性能调优

### 大型照片库（50K+ 张）

```bash
# 增加 Postgres 共享缓冲区
# 在 docker-compose.yml 的 database 服务下加：
command: postgres -c shared_buffers=512MB -c effective_cache_size=2GB
```

### 上传慢

```bash
# 增加上传限制（在 .env 里）
UPLOAD_LIMIT=10000  # 10GB，4K 视频用
```

### 低内存系统（4GB）

- 关闭机器学习（`IMMICH_MACHINE_LEARNING_ENABLED=false`）
- 只用缩略图生成（关闭 CLIP）
- Postgres 设置为 256MB shared_buffers

### 监控

```bash
# 查看资源占用
docker stats

# 查看日志
docker compose logs -f immich_server
docker compose logs -f immich_machine_learning
```

## 常见坑与解决方案

### 1. 手机端"上传失败"

**原因：** 默认上传限制是 300MB。4K 视频超了。

**解决：** 在 `.env` 里设置 `UPLOAD_LIMIT=10000`，然后 `docker compose restart`。

### 2. 人脸没检测到

**原因：** ML 容器没跑起来或内存不够。

**解决：**
```bash
docker compose ps  # 检查 ML 容器状态
docker compose logs immich_machine_learning  # 看有没有 OOM 错误
```

如果是 OOM：加内存或暂时关掉 ML。

### 3. 外部库不显示

**原因：** 卷挂载路径不对。

**解决：** 确保 `docker-compose.yml` 里的路径和你实际的存储位置一致：
```yaml
volumes:
  - /mnt/photos:/usr/src/app/upload
```

而且这个路径要让容器用户有访问权限。

### 4. PostgreSQL 连接被拒绝

**原因：** 数据库容器没启动成功（通常是密码不匹配）。

**解决：** 检查 `.env` 和 Postgres 期望的是否一致。如果搞乱了：
```bash
docker compose down -v  # 警告：会删除数据库
docker compose up -d    # 重新来过
```

### 5. 缩略图看起来模糊

**原因：** 低质量缩略图生成。

**解决：** 不是 bug——Immich 会生成多种尺寸。"模糊"的预览先加载，完整质量的稍后加载。去设置 → 缩略图配置检查。

## 备份策略

**3-2-1 原则：** 3 份副本，2 种不同介质，1 份异地。

```bash
# 简单的 rsync 备份到外部硬盘
rsync -av --progress /path/to/immich/upload/ /mnt/backup/immich/

# 用 cron 自动化（每天凌晨 3 点）
0 3 * * * rsync -av /path/to/immich/upload/ /mnt/backup/immich/
```

数据库备份：
```bash
# $DB_USERNAME 来自你的 .env（默认：postgres）
docker exec immich_postgres pg_dump -U "$DB_USERNAME" immich > immich_db_backup.sql
```

> **坑 #3：** 别只备份数据库——`/upload` 目录也要备份。数据库存的是元数据，你的实际照片在 `/upload` 里。

## FAQ

### 能从 Google Photos 迁移到 Immich 吗？

能。用 [Immich CLI](https://github.com/immich-app/immich-cli) 通过 API 上传，或者用手机 App 重新上传（你的手机里本来就有照片）。没有自动的 Google Photos 导入——你需要先从 Google Takeout 导出。

### Immich 支持 RAW 照片吗？

支持。它能处理 RAW 文件并生成预览。Lightroom DNG、Canon CR2/CR3、Sony ARW、Nikon NEF——全支持。

### 能和非 Immich 用户分享相册吗？

能。公开分享链接，可选密码保护。查看不需要账号。

### Immich 和 PhotoPrism 有什么区别？

| | Immich | PhotoPrism |
|--|--------|------------|
| **移动端** | 优秀（原生 App） | 不错（PWA） |
| **自动备份** | 有（后台运行） | 无（手动） |
| **ML 模型** | 更多（人脸、CLIP、重复检测） | 较少（仅人脸） |
| **界面** | 类 Google Photos | 更偏桌面端 |
| **社区** | 60K+ Stars | 35K+ Stars |

**我的看法：** 要 Google Photos 替代品就选 Immich。要照片库管理器就选 PhotoPrism。

### 我到底需要多少内存？

- **4GB：** 关掉 ML，存储和查看体验依然很好
- **8GB：** 完整 ML 功能，50K+ 张照片流畅运行
- **16GB+：** 除非你跑 100K+ 张照片且全开 ML，否则用不上

### 能在反向代理后面用 Immich 吗？

能。看[官方反向代理文档](https://immich.app/docs/install/reverse-proxy)。Nginx、Caddy、Traefik、Coolify 都支持。

不想折腾反向代理？用 [Tailscale + Headscale](/blog/tailscale-headscale-guide-zh_CN) 搭零配置 VPN，手机和电脑直接访问 Immich，无需暴露端口到公网。

### 该选哪个：Immich vs PhotoPrism vs Lychee？

| 使用场景 | 推荐 |
|----------|---------------|
| **Google Photos 替代品** | **Immich** — 原生手机备份、智能搜索、人脸识别 |
| **低资源硬件上的照片库** | **Lychee** — 512MB 内存就能跑，极简 |
| **桌面端照片库管理器** | **PhotoPrism** — 界面干净，元数据工具不错 |
| **已经在用 Nextcloud** | **Nextcloud Photos** — 不用再加一套技术栈 |
| **隐私优先、功能极简** | **Lychee** — 攻击面最小 |

**一句话：** Immich 功能最全。PhotoPrism 最均衡。Lychee 最轻。如果你已经有 Nextcloud，先用它的照片应用，别再加一个服务。
